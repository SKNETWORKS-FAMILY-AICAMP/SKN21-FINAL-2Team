# 장소 검색(Retrieval) 동작 원리

> 관련 파일: `backend/app/core/retrieval/place.py`, `backend/app/core/retrieval/place_score.py`, `backend/app/agents/retriever.py`

---

## 개요

사용자가 "홍대 분위기 좋은 카페"라고 입력하거나 사진을 첨부하면, 시스템은 단순히 단어를 검색하는 것이 아니라 **여러 방법으로 동시에 검색한 뒤 점수를 합산**하여 가장 적합한 장소를 찾아냅니다.

이 과정은 크게 **3단계**로 이루어집니다.

```
1단계: 입력 유형에 따라 여러 채널로 후보 장소 수집
        ↓
2단계: 후보 장소마다 보너스 점수 보정
        ↓
3단계: AI가 최종 순위 결정 (Reranker)
```

---

## 검색 유형: 일반 검색 vs 여행 일정 계획

시스템은 사용자의 의도를 먼저 파악한 뒤, **두 가지 검색 방식** 중 하나 또는 둘 다를 실행합니다.

### 일반 검색 (기본)
"홍대 감성 카페 추천해줘"처럼 특정 장소를 찾는 경우입니다.

```
사용자 입력 한 문장
    → 하이브리드 검색 1회 실행
    → 후보 장소 풀 생성
```

### 여행 일정 계획 (TRIP_PLANNING)
"제주도 2박 3일 여행 일정 짜줘"처럼 여행 계획을 요청하는 경우입니다.

```
Planner AI가 일정 초안 작성
    → 일정 항목별로 개별 검색 (최대 3개 병렬)
    → 예: "제주 아침 식사 맛집", "성산일출봉 근처 관광지", ...
    → 전체 후보를 합산하여 각 일정에 맞는 장소 배정
    → 결과가 0개일 때만 일반 검색으로 자동 fallback
```

> **핵심 차이**: 일반 검색은 하나의 쿼리로 한 번 검색하고, 여행 일정 계획은 일정 항목 수만큼 검색을 반복하여 각 시간대에 맞는 장소를 개별적으로 찾습니다. **두 검색이 동시에 실행되지 않습니다.** TRIP_PLANNING이면 여행 일정 검색만, 그 외 의도면 일반 검색만 실행됩니다.

## 입력 유형별 검색 채널

사용자가 텍스트를 입력했는지, 이미지를 첨부했는지에 따라 **작동하는 채널이 달라집니다.**

### 케이스 1. 텍스트만 입력한 경우

```
"홍대 감성 카페 추천"
    │
    ├─→ [채널 A] BGE-M3 의미 검색 ─────────→ 장소 DB 검색    (가중치 × 1.0)
    │            "분위기", "카페"의 뜻을 이해
    │
    ├─→ [채널 B] Sparse 단어 검색 ─────────→ 장소 DB 검색    (가중치 × 0.85)
    │            "홍대"라는 단어가 정확히 있는 곳 (Qdrant Native Sparse Vector 사용)
    │
    └─→ [채널 C] CLIP 텍스트→이미지 검색 ──→ 사진 DB 검색    (가중치 × 0.5)
                 감성 카페 분위기의 사진이 있는 장소
```

### 케이스 2. 이미지만 첨부한 경우

```
📷 사진 첨부 (텍스트 없음)
    │
    ├─→ [채널 D] CLIP 시각 유사 검색 ──────→ 사진 DB 검색    (가중치 × 1.0)
    │            사진과 분위기가 비슷한 장소 사진 탐색
    │
    └─→ [보조]   GPT-4o-mini가 사진 분위기를 글로 설명
                 "따뜻한 조명의 아늑한 카페, 빈티지 인테리어..."
                    │
                    └─→ [채널 D보조] BGE-M3 의미 검색 ──→ 장소 DB  (가중치 × 0.8)
```

### 케이스 3. 텍스트 + 이미지 동시 입력

```
"이런 분위기 카페 홍대에서 찾아줘" + 📷 사진
    │
    ├─→ [채널 A] BGE-M3 의미 검색 ─────────→ 장소 DB    (× 1.0)
    ├─→ [채널 B] Sparse 단어 검색 ─────────→ 장소 DB    (× 0.85)
    ├─→ [채널 C] CLIP 텍스트→이미지 검색 ──→ 사진 DB    (× 0.5)
    ├─→ [채널 D] CLIP 시각 유사 검색 ──────→ 사진 DB    (× 1.0)
    └─→ [채널 D보조] 감정 텍스트 의미 검색 → 장소 DB    (× 0.8)
```

### 채널 E. BM25 단어 빈도 검색 (조건부 실행)

텍스트 입력이 있고, 위 채널 결과가 **너무 적거나 점수가 낮을 때만** 추가로 실행됩니다.

- **비유**: 장소 설명에 검색어 단어가 얼마나 자주 등장하는지 세는 방식
- **가중치**: `× 0.7`

## 위치 기반 필터 (Geo Filter)

"홍대", "강남역", "명동" 같은 **알려진 지역명**이 검색어에 포함되면, 해당 지역 반경 내의 장소만 먼저 걸러냅니다.

```
"홍대 카페" 입력
    │
    → 시스템: "홍대" → 위도 37.5547, 경도 126.9230, 반경 1,200m로 변환
    → 장소 DB에서 이 반경 안에 있는 장소만 검색 대상에 포함
    → 반경 내 결과가 0개이면 → 필터 없이 재검색 (자동 fallback)
```

> 위치 필터는 장소 DB에만 적용됩니다. 사진 DB에는 위치 정보가 없으므로 사진 검색은 필터 없이 실행됩니다.

현재 등록된 지역 랜드마크 (12개):

| 표준명 | 대표 별칭 |
|:---|:---|
| 홍대 | 홍대입구, 홍대거리, 홍익대, 홍대앞 |
| 건대 | 건국대, 건대입구, 화양동 |
| 가로수길 | 신사동가로수길 |
| 샤로수길 | 서울대입구, 서울대입구역 |
| 송리단길 | 석촌호수 맛집, 송파동 카페거리 |
| 서촌 | 세종마을, 통의동, 체부동, 경복궁 서측 |
| 북촌 | 북촌한옥마을, 가회동, 삼청동, 계동 |
| 성수동 | 연무장길, 성수핫플, 서울숲 카페거리 |
| 망리단길 | 망원시장, 망원동 카페거리 |
| 을지로 | 힙지로, 을지로3가, 을지로4가 |
| 경리단길 | 이태원 경리단, 회나무로 |
| 해방촌 | HBC, 용산동2가, 신흥시장 |

## 점수 합산 방식 (RRF: Reciprocal Rank Fusion)

각 채널의 결과를 **순위 기반**으로 합산합니다. 1위일수록 더 높은 점수, 순위가 낮을수록 점수가 작아집니다.

```
채널 기여 점수 += 채널_가중치 × (1 / (60 + 해당_채널_순위))
```

예시: 채널 A에서 1위, 채널 B에서 3위인 장소
```
점수 += 1.0 × (1 / 61) ≈ +0.0164   ← 채널 A 기여
점수 += 0.85 × (1 / 63) ≈ +0.0135  ← 채널 B 기여
합계 ≈ 0.0299
```

같은 장소가 여러 채널에서 등장할수록 점수가 누적되어 상위로 올라갑니다.

## 2단계: 점수 보정 (Boost)

1단계에서 나온 후보 장소 각각에 대해 **4가지 보너스 점수**를 추가로 계산합니다.

```
boost 합계 = 키워드 보너스 + 지역 텍스트 보너스 + 거리 보너스 + 주소 토큰 보너스

score = first_stage_score + BOOST_WEIGHT(0.3) × boost 합계
```

> **BOOST_WEIGHT(0.3) 적용 이유**: RRF `first_stage_score` 범위(0.010~0.050)에 비해 boost 합계(최대 0.45)가 지나치게 크면 의미론적 유사도 점수가 무시됩니다. BOOST_WEIGHT로 스케일을 보정하여 RRF 점수와 boost가 균형 있게 반영됩니다. (boost 최대 기여: 0.45 × 0.3 ≈ 0.135)

---

### 보너스 1. 키워드 매칭 보너스
**"사용자가 특정 상호명을 직접 언급했는가?"**

| 조건 | 점수 |
|:---|:---|
| 검색어 토큰 중 장소 이름이 **정확히 일치** (3자 이상, 예: "봉피양 예약" → 상호명 "봉피양") | +0.18 |
| 검색어 토큰과 장소 이름 토큰이 **일부 겹침** (4자 이상 상호명 한정) | +0.02 × 겹친 수 (최대 +0.06) |

**최대 보너스: +0.15** (cap 적용)

> ⚠️ **오매칭 방지**: "홍대 김밥" 검색 시 강남의 "홍대김밥" 상호가 부당하게 유리해지는 것을 막기 위해, 토큰 단위 완전 일치만 허용합니다. 행정구역 매칭은 보너스 4(주소 토큰)가 전담합니다.

---

### 보너스 2. 지역 텍스트 보너스
**"사용자가 원하는 지역명이 장소의 주소나 이름에 포함되어 있는가?"**

- 입력: `preferred_location` (예: "마포구", "홍대 근처")
- 지역 토큰이 장소 주소/이름과 매칭될수록 가산

| 조건 | 점수 |
|:---|:---|
| 지역 토큰이 장소 주소/이름 토큰에 일치 | +0.03 × 매칭 수 |

**최대 보너스: +0.10**

---

### 보너스 3. 물리적 거리 보너스
**"사용자/기준 위치에서 얼마나 가까운가?"**

- 입력: 사용자 GPS 좌표, 또는 랜드마크 좌표 (GPS 없을 때 폴백)
- Haversine 공식으로 실제 거리(km) 계산 후 반경 **10km** 기준 선형 감소 (`GEO_PROXIMITY_RADIUS_KM=10.0`)

```
거리 보너스 = 0.20 × (1 - 거리km / 10km)
```

| 거리 | 보너스 |
|:---|:---|
| 0km (동일 위치) | +0.20 |
| 5km | +0.10 |
| 10km 이상 | 0 |

**최대 보너스: +0.20**

---

### 보너스 4. 주소 토큰 보너스
**"검색어의 행정구역 단어가 장소 주소에 정확히 들어있는가?"**

> ⚠️ **현재 비활성화** (`ENABLE_ADDR_SPARSE_BOOST = False`): addr_sparse_boost는 항상 0.0으로 계산됩니다. 아래는 활성화 시 동작 방식입니다.

- 검색어에서 주소성 단어 추출 (예: "마포구", "홍대") — **선호 지역(`preferred_location`)은 제외, 보너스 2가 전담**
- 장소의 미리 저장된 주소 토큰(`addr_tokens`)과 비교

| 조건 | 가중치 |
|:---|:---|
| 정확히 일치 (예: "홍대" == "홍대") | `SPARSE_ADDR_EXACT_WEIGHT` (0.04) |
| 어간 일치 (예: "마포구" → "마포") | `SPARSE_ADDR_STEM_WEIGHT` (0.02) |

**최대 보너스: `SPARSE_ADDR_MAX_BOOST` (0.20, 비활성 시 0)**

## 3단계: 최종 순위 결정 (Reranker)

1~2단계에서 걸러진 **상위 후보 장소들(최대 8개)**을 AI 모델이 다시 한번 평가합니다.

- **모델**: `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1` (다국어 CrossEncoder)
- **방법**: 검색어와 장소 정보(이름 + 카테고리 + 주소)를 쌍으로 넣어 **관련도 점수** 계산
- **결과**: 이 최종 점수(`rerank_score`) 기준으로 재정렬 후 최종 결과 반환

> Reranker를 로드하지 못한 경우 2단계 점수 기준으로 그대로 반환합니다.

---

## 4단계: 답변 생성 및 ID 매핑 (ID Tagging)

검색된 장소 정보를 기반으로 Executor Agent가 최종 답변을 생성할 때, UI에서 해당 장소를 정확히 매핑할 수 있도록 **ID 태깅** 시스템을 사용합니다.

### ID 추출 방식
- **태깅 매칭**: LLM이 답변 끝에 `[IDs: id1, id2, ...]` 형식으로 선택한 장소의 ID를 명시합니다.
- **텍스트 추론 (Fallback)**: 태그가 없거나 유효하지 않은 경우, LLM의 답변 본문에서 장소 이름을 추출하여 검색 후보군과 매칭해 ID를 찾아냅니다.
- **검증**: 추출된 ID가 실제 검색된 후보군(`candidate_pool`)에 존재하는지 최종 확인 후 유효한 ID만 `selected_ids`로 본문에 포함시킵니다.

### 멀티모달 답변 (Vision)
사용자가 이미지를 첨부한 경우, Executor는 텍스트 컨텍스트뿐만 아니라 **이미지 원본(base64)**을 LLM에 함께 전달하여, "이미지의 분위기와 어울리는 장소"를 직접 판단하여 답변을 생성합니다.

### 웹 검색 보안 (Tavily Fallback)
장소 DB에서 결과가 전혀 없는 경우, Tavily API를 통해 웹 검색을 수행합니다. 이때 단순 검색어뿐만 아니라 사용자의 **지역 정보(location slots)**를 조합하여(예: "마포구 여행 맛집") 더 정확한 정보를 찾아냅니다.

## 점수 구성 및 실제 평균 범위

검색 결과 각 항목에는 3가지 계층의 점수가 기록됩니다.

### 점수 계층 구조

```
first_stage_score   ← 1단계: 채널 RRF 합산만
      +
score_boost_total   ← 2단계: 4가지 보너스 합계
      =
score               ← 1+2단계 합산 (후보 선별 기준)
      ↓
rerank_score        ← 3단계: AI 재순위 점수 (최종 정렬 기준)
```

### 실제 점수 범위

모든 점수는 **[0.0, 1.0]** 범위로 정규화되어 반환됩니다.

| 점수 종류 | 정규화 범위 | 정규화 방법 | 설명 |
|:---|:---|:---|:---|
| `first_stage_score` | 0.0 ~ 1.0 | `min(1.0, raw / RRF_SCORE_MAX(0.08))` | 채널 RRF 합산. 단일 채널 1위 raw ≈ 0.016 → 정규화 ≈ 0.21 |
| `score_boost_total` | 0.0 ~ 1.0 | `min(1.0, raw / MAX_BOOST_SUM(0.65))` | raw boost 합계. addr_sparse 비활성 시 실제 최대 raw = 0.45 → 정규화 ≈ 0.69 |
| `score` | 0.0 ~ 1.0 | `min(1.0, raw / FUSED_SCORE_MAX(0.20))` | `first_stage + 0.3 × boost`. raw 최대 ≈ 0.185 → 정규화 ≈ 0.93 |
| `rerank_score` | 0.0 ~ 1.0 | `sigmoid(logit)` | CrossEncoder 관련도. 높음 ≈ 0.27~0.73 / 낮음 ≈ 0.00~0.05 |

**정규화 기준값** (환경변수로 튜닝 가능):
- `RRF_SCORE_MAX=0.08` — 모든 채널이 동시에 1위인 이론적 RRF 최대값
- `FUSED_SCORE_MAX=0.20` — score 이론적 최대값 (RRF + BOOST_WEIGHT × boost)
- `MAX_BOOST_SUM=0.65` — 모든 보너스 활성 시 raw boost 이론적 최대합

> **최종적으로 사용되는 점수**: `rerank_score`
> 1~2단계 `score`는 후보를 추리는 데 사용하고, 실제 최종 순서는 reranker가 결정합니다.
> query가 없거나(이미지 전용에서 emotional_text도 없는 경우) reranker 실패 시 → `rerank_score = None`, `score` 순 유지.

**예시** ("홍대 감성 카페 추천" 검색 후 정규화된 점수):
```
홍대 실제 카페    rerank_score ≈ 0.82   → 1위 ✅  (logit -1.5 → sigmoid)
연남동 카페       rerank_score ≈ 0.04   → 2위     (logit -3.2 → sigmoid)
경복궁 관광지     rerank_score ≈ 0.00   → 최하위 ✅ (logit -7.3 → sigmoid)
```

## 최종 후보 선별 방식

Reranker까지 거친 결과에서 **카테고리 다양성**을 고려하여 최종 5개를 선택합니다.

```
결정론 모드 (기본):
  1순위: 카테고리가 겹치지 않는 장소를 점수 순으로 선택
  2순위: 남은 슬롯은 점수 순으로 채움

탐색 모드 (Explore):
  점수를 가중치로 쓴 랜덤 선택 → 매번 다른 결과 반환 가능
```

## 검색 결과에 포함되는 점수 정보

검색 결과 각 항목에는 디버깅을 위해 다음 점수 정보가 포함됩니다:

| 필드 | 설명 |
|:---|:---|
| `score` | 최종 1차 점수 (채널 합산 + Boost 합계) |
| `first_stage_score` | Boost 적용 전 채널 합산 점수만 |
| `keyword_match_boost` | 키워드 보너스 |
| `location_text_boost` | 지역 텍스트 보너스 |
| `geo_proximity_boost` | 거리 보너스 |
| `addr_sparse_boost` | 주소 토큰 보너스 |
| `score_boost_total` | Boost 합계 |
| `rerank_score` | 3단계 AI 재순위 점수 **(최종 정렬 기준)** |
| `final_rank` | 최종 순위 |
| `match_types` | 매칭된 채널 목록 (예: `["text_semantic", "addr_sparse"]`) |

## 자주 묻는 질문 (FAQ)

**Q. "홍대 김밥"을 검색했는데 강남에 있는 "홍대김밥" 식당이 왜 상위에 나오지 않나요?**
A. 의도적으로 그렇게 설계되어 있습니다. 키워드 보너스는 토큰 단위 완전 일치만 허용하고, 지역 필터가 홍대 반경 내 장소에만 적용되므로, 강남의 주소는 주소 토큰 보너스에서도 점수를 받지 못합니다.

**Q. 사진을 첨부하면 어떻게 달라지나요?**
A. 사진의 시각적 특징을 CLIP이 분석해 비슷한 분위기의 사진 DB를 검색하고, GPT-4o-mini가 사진 분위기를 글로 변환하여 텍스트 검색도 추가로 실행합니다. 텍스트와 이미지 결과가 함께 합산됩니다.

**Q. 위치 정보를 켜면 어떻게 달라지나요?**
A. GPS 좌표를 기반으로 가까운 장소에 거리 보너스(최대 +0.20)가 추가됩니다. 또한 역지오코딩으로 현재 주소를 자동으로 검색어에 포함시킵니다.

**Q. "홍대" 같은 지역명을 입력하면 어떻게 처리되나요?**
A. 내부적으로 "홍대" → 위도/경도/반경으로 변환하여, 해당 반경 안에 있는 장소만 먼저 걸러냅니다. 결과가 없으면 반경 필터 없이 자동으로 재검색합니다.

**Q. rerank_score는 어떻게 해석하나요?**
A. mMiniLMv2 모델의 raw logit에 sigmoid를 적용하여 **0.0~1.0** 범위로 정규화됩니다. 0에 가까울수록 관련도가 낮고, 1에 가까울수록 관련도가 높습니다. 실제 검색에서는 관련 있는 장소가 0.2~0.8, 관련 없는 장소가 0.0~0.05 범위에 분포합니다. **절대값보다 상대적 순서가 중요**합니다.

**Q. 여행 일정 계획과 일반 검색의 결과가 왜 다른가요?**
A. 여행 일정 계획은 Planner AI가 먼저 "아침 식사 → 오전 관광 → 점심 → 오후 체험" 같은 일정 초안을 만들고, 각 항목에 맞는 장소를 **별도로 검색**합니다. 일반 검색보다 맥락에 맞는 장소가 배정되지만, 검색 횟수가 많아 처리 시간이 더 걸립니다.
